## Purpose

Exit criteria for FT-011 (TypeScript AppHost parity — generated
module mirroring `WithCoolifyDeploy` / `WithImageRegistry` /
`WithCoolifyDestination` / `WithManagedDashboard`). Pins the
emitted-module's public contract (I-1), the pure-recording-shim
property (I-2), the per-function idempotency rules (I-3), the
ADR-001 invariants-apply-identically guarantee (I-4), the no-handle-
value-read property (I-5), the runtime-backstop null/undefined
discipline (I-6), the credentials-tuple shape enforcement (I-7), the
byte-stable emitted-file contract (I-9), and the no-TS-side-
reimplementation-of-phases discipline (I-11).

## Fixture

A minimal TypeScript Aspire AppHost project that references the
Coolify publisher assembly, scaffolded by `aspire restore` after the
publisher is wired into the project's dependency graph (the same
mechanism `aspire-ssh-deploy`'s TS variant uses). The C# publisher
(FT-001 / FT-003 / FT-005 / FT-010) is assumed implemented; FT-011's
generator hook is registered with `aspire restore`.

A representative C# AppHost with equivalent registration data is
available for cross-validation (assertion D).

## Assertions

### A. `aspire restore` emits the module (FT-011 §Behaviour §0)

1. After `aspire restore` runs on the fixture project, exactly one
   TypeScript file appears in the project's generated-modules
   directory (the conventional Aspire-TS path). The file's first
   line is the literal
   `// auto-generated by aspire restore; do not edit`.
2. Filesystem-diff of `aspire restore` invocation shows the
   generator wrote nowhere outside the designated
   generated-modules directory (FT-011 I-10).

### B. Public contract — exactly four named exports (FT-011 I-1,
   I-12)

3. The emitted module exports exactly four named functions:
   `withCoolifyDeploy`, `withImageRegistry`,
   `withCoolifyDestination`, `withManagedDashboard`. Asserted by
   parsing the module's exports table.
4. The module declares no default export.
5. The signatures of the four functions match FT-011 §Behaviour:
   - `withCoolifyDeploy(builder, url, token)` — all three
     required, `url` and `token` typed as parameter-handle
     references;
   - `withImageRegistry(builder, prefix, credentials?)` —
     `credentials` is an optional two-element tuple of
     parameter-handle references;
   - `withCoolifyDestination(builder, name)` — both required;
   - `withManagedDashboard(builder, dashboardToken)` — both
     required, `dashboardToken` typed as a parameter-handle
     reference.

### C. Pure recording shim (FT-011 I-2, I-11)

6. Source-level inspection of the emitted module shows each
   function body contains only argument validation and a single
   slot-write call to the publisher instance. Zero `fetch(`,
   zero `import(\"http\")`, zero `child_process`, zero `docker`
   string literal, zero image-tag composition, zero
   phase-name string literals (`\"configure\"`, `\"build\"`,
   `\"push\"`, `\"deploy\"`, `\"verify\"`) appear in the module
   body.

### D. TS AppHost produces identical Coolify-side shape to a C#
   equivalent (FT-011 I-4)

7. A TS AppHost calling the four functions with parameter handles
   carrying the same values as a representative C# AppHost,
   `aspire deploy`-ed against the same Coolify instance, produces
   byte-identical Coolify-side state (project, environment,
   services, env-vars, dashboard) compared with the C# AppHost
   deployed against a freshly-reset Coolify instance. ADR-001
   invariants apply identically.

### E. Per-function idempotency rules (FT-011 I-3, I-8)

8. **`withCoolifyDeploy` — first-call-wins.** Calling it twice
   with different `(url, token)` pairs leaves the first pair in
   the publisher slot; the second call is a silent no-op. (No
   exception is thrown.) Asserted by inspecting the publisher
   slot after both calls.
9. **`withImageRegistry` — last-call-wins.** Calling it twice
   with different triples leaves the second triple in the slot.
10. **`withCoolifyDestination` — last-call-wins.**
11. **`withManagedDashboard` — last-call-wins on the handle;
    opt-in flag remains true.** Calling it zero times leaves
    the flag false (silent opt-out — FT-010 I-6 inherited).

### F. No handle value read by the shim (FT-011 I-5)

12. Injecting sentinel values into the TS AppHost's parameter
    handles (`SENTINEL_TS_TOKEN_DO_NOT_LEAK_011`, etc.) and
    inspecting all log lines emitted by FT-011's shim functions,
    the publisher slot's debug surface, and any stderr produced
    during the four shim calls returns zero matches for the
    sentinel literals. The shim never invokes a value-reading
    API on the handle type.

### G. Runtime backstop on null / undefined (FT-011 I-6)

13. Calling each shim function from un-typechecked JS with
    explicit `null` or `undefined` for any required argument
    throws a TS `Error` whose `.message` names the offending
    argument and points back to the corresponding C# extension
    (e.g.
    `withCoolifyDeploy: 'token' argument must be a parameter handle; got undefined. See FT-001.`).

### H. `withImageRegistry` credentials tuple shape enforcement
   (FT-011 I-7)

14. Calling `withImageRegistry(builder, prefix, [user, null])`
    from un-typechecked JS throws naming `credentials[1]`.
    Symmetric for `credentials[0]` null.
15. TypeScript's compile-time type for the `credentials` tuple
    rejects a one-element tuple at compile time (asserted by a
    TS compilation-error fixture that attempts
    `withImageRegistry(builder, prefix, [user])`).

### I. Byte-stable emitted file (FT-011 I-9)

16. Running `aspire restore` twice on the same project against the
    same publisher version produces byte-identical generated
    files (SHA-256 comparison). No timestamps, no machine paths,
    no temp-dir derivations leak into the file body.

### J. No new error symbols introduced (FT-011 §"Out of scope")

17. Grepping the emitted module for `E_` and `W_` symbol-shaped
    string literals returns zero matches. The TS shim does not
    introduce stable observable error symbols beyond the
    standard TS `Error` exceptions in G / H.

### K. Recorded data is written to the same publisher slots
   the C# extensions write to (FT-011 I-2 tail)

18. After a TS AppHost calls all four shim functions and
    `aspire deploy` runs, the underlying .NET publisher reads
    the captured `(url, token)` / `(prefix, user, pass)` /
    destination / dashboard-token handles from the same
    publisher-instance slots a C# AppHost would write to. There
    is no second registration channel and no
    TS-specific publisher-state object.

### L. Ordering policy mirrors C# (FT-011 §"Error handling")

19. Whatever the C# extensions do when chained off the builder
    rather than off the `WithCoolifyDeploy(...)` result, the TS
    shim does the same. Asserted by parity: a TS test calling
    `withImageRegistry` before `withCoolifyDeploy` produces the
    same outcome (either silently attaches the publisher, or
    throws a precise "call withCoolifyDeploy first" error) as
    the C# equivalent does.

## Pass criteria

All nineteen assertion groups pass deterministically across at
least three consecutive runs. Any sentinel leak detected in F §12
is a hard fail.

## Out of scope

- TS-side reimplementation of phases / Coolify client / image
  flow (forbidden by I-2 / I-11).
- New error symbols introduced by FT-011 (forbidden by §"Out of
  scope").
- Multi-target `withCoolifyDeploy` on a single TS builder
  (first-call-wins applies; multi-target is post-v1 for both
  surfaces).
- Hand-written / developer-customised module (not supported).
- Non-TypeScript authoring surfaces (Python, F#, etc.).
- `aspire restore`'s generator-modules infrastructure itself
  (Aspire SDK concern).
